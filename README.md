# Vehicle-type-detector  
Проект студентов 3 курса ИИР НГУ  

---

## 📌 Описание задачи

Целью данного проекта является проведение комплексного анализа датасета изображений транспортных средств для выявления:

- закономерностей распределения данных,
- аномалий и потенциальных проблем,
- факторов, способных повлиять на обучение модели детекции.

Результаты анализа используются для:

- понимания структуры данных,
- оценки качества и сбалансированности датасета,
- выявления источников bias,
- выбора стратегий обучения модели.

---

## 🔍 Проведённый анализ

В рамках исследования были рассмотрены следующие аспекты:

- распределение объектов по классам;
- анализ дубликатов изображений;
- распределение объектов по типам сцен (single-object / multi-object);
- взаимосвязь между классами и контекстом сцены;
- анализ цветовых характеристик изображений;
- анализ размеров и геометрии bounding box (bbox);
- корреляционный анализ признаков сцены и классов объектов.

---

## 📊 Основные результаты

---

### 1. Class Imbalance

В датасете наблюдается сильный дисбаланс классов:

- `car` значительно доминирует (~44k объектов),
- `personal mobility` занимает второе место,
- ряд классов (`quadbike`, `emergency`, `construction equipment`) представлены слабо.

#### ❗ Риски
- модель склонна переобучаться на доминирующие классы;
- снижение качества распознавания редких классов.

#### ✅ Возможные решения
- расширение датасета (сбор или генерация данных);
- oversampling редких классов;
- использование весов классов в функции потерь;
- применение Focal Loss (частично реализовано в современных версиях YOLO).

---

### 2. Scene-Conditioned Imbalance

Обнаружена зависимость между количеством объектов на изображении и классами:

- `car` чаще встречается в multi-object сценах;
- `personal mobility` — в одиночных сценах.

Корреляции:
- `num_objects ↔ car ≈ +0.67`
- `num_objects ↔ personal mobility ≈ -0.23`

#### ❗ Риски
Модель может выучить ложные зависимости:
- "много объектов → car"
- "один объект → personal mobility"

#### ✅ Возможные решения
- добавление multi-object сцен с редкими классами;
- добавление single-object сцен с классом `car`;
- использование аугментаций:
  - Mosaic
  - MixUp
  - Copy-Paste

---

### 3. Context Bias

Следствие предыдущей проблемы: модель может использовать контекст сцены вместо признаков объекта.

Пример:
> объект класса `personal mobility` среди автомобилей может классифицироваться как `car`

#### ❗ Риски
- плохая обобщающая способность;
- ошибки в нетипичных сценах.

#### ✅ Решения
- размывание контекста через аугментации;
- генерация контрпримеров (объекты в нетипичных сценах);
- random crop, copy-paste.

---

### 4. Color Bias

Выявлены зависимости между яркостью сцены и классами:

- `brightness ↔ personal mobility ≈ +0.31`
- `brightness ↔ car ≈ -0.18`

#### ❗ Вывод
Цвет сцены частично коррелирует с классом объектов.

#### ❗ Риск
Модель может использовать освещение как косвенный признак.

---

### 5. Bounding Box Analysis

Основные характеристики bbox:

- медианная площадь: ~0.003 (очень маленькие объекты)
- средняя площадь: ~0.049
- aspect ratio: среднее ≈ 0.85, max = 10

#### ❗ Выводы

1. **Small Object Bias**
   - большинство объектов занимает крайне малую часть изображения;
   - особенно выражено для класса `car`.

2. **Geometric Bias**
   - размеры объектов различаются между классами:
     - `car` → маленькие;
     - `personal mobility` → крупнее;
     - `truck` → средние/крупные.

3. **Экстремальные значения**
   - встречаются bbox, покрывающие почти всё изображение;
   - возможны ошибки разметки.

---

### 6. Distance Bias

Малые объекты (например, `car`) часто находятся на дальнем плане.

#### ❗ Риски
- ухудшение детекции удалённых объектов;
- увеличение числа пропусков (false negative).

---

## 🧠 Общий вывод

Датасет является пригодным для обучения модели, однако содержит ряд значимых смещений:

- выраженный классовый дисбаланс;
- зависимость классов от типа сцены;
- различия в размерах объектов между классами;
- наличие small object bias;
- частичная зависимость от цветовых характеристик сцены.

Это может привести к тому, что модель будет:

- использовать контекст сцены вместо признаков объекта;
- хуже распознавать редкие классы;
- ошибаться на нестандартных сценах.

## 🚀 Рекомендации

Для повышения качества модели рекомендуется:

- балансировка датасета (data-level);
- использование продвинутых аугментаций:
  - Mosaic
  - MixUp
  - Copy-Paste
- применение loss-level методов:
  - class weights
  - focal loss
- дополнительная проверка bbox-разметки;
- усиление обучения на малых объектах.
